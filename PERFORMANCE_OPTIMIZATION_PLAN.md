# 파일 전송 성능 최적화 설계 문서

## 📊 현재 상태 분석

### 현재 구현 사항
- **청크 크기**: 128KB (동적 조정)
- **전송 채널**: 단일 WebRTC 데이터 채널
- **혼잡 제어**: TCP AIMD (Additive Increase Multiplicative Decrease)
- **저장 방식**: OPFS 동기 쓰기 (SyncAccessHandle)
- **순서 보장**: 단일 채널로 순서 보장

### 성능 지표
- 현재 처리량: 약 10-20 MB/s (일반적인 네트워크)
- 병목지점: WebRTC 데이터 채널 처리량 제한 및 메모리 버퍼 관리

---

## 🚀 성능 최적화 전략 (우선순위 순)

### [1] 멀티 채널 병렬 전송 (🔴 높은 우선순위)

**현재 문제점:**
- 단일 채널로 순서대로 전송하면 한 번에 하나의 청크만 처리
- 네트워크 왕복 지연(RTT)만큼 성능 저하

**해결 방안:**
- 4-8개의 독립적인 WebRTC 데이터 채널 생성
- Sender Worker에서 라운드 로빈 방식으로 청크 분배
- Receiver Worker에서 청크 시퀀스 추적 및 조립
- 채널별 ACK 관리

**기대 효과:**
- 처리량: 약 **2-4배 향상** (채널 수에 따라)
- 약 40-80 MB/s 달성 가능

**구현 난이도:** ⭐⭐⭐ (중간)

**리스크:**
- 브라우저 메모리 사용량 증가 (채널당 ~10-20MB 버퍼)
- 채널 관리 복잡도 증가

---

### [2] 동적 청크 크기 최적화 (🟠 높은 우선순위)

**현재 문제점:**
- 고정 청크 크기로 인한 네트워크 이용률 저하
- 네트워크 상태에 따른 즉각적인 적응 불가

**해결 방안:**
```
네트워크 상태 모니터링:
  - RTT (Round Trip Time) 측정
  - 패킷 손실률 추적
  - 버퍼 포화도 확인

동적 조정:
  - RTT < 50ms: 256KB
  - RTT 50-150ms: 128KB
  - RTT > 150ms: 64KB
  - 패킷 손실 > 5%: 32KB로 감소
```

**기대 효과:**
- 처리량: **15-30% 향상**
- 네트워크 환경에 자동 적응

**구현 난이도:** ⭐⭐ (낮음)

**구현 위치:**
- `file-sender.worker.v2.ts` - ACK 메시지에서 RTT 측정
- `constants.ts` - 동적 임계값 조정

---

### [3] OPFS 쓰기 최적화 (🟠 높은 우선순위)

**현재 문제점:**
- 동기 쓰기(Sync)로 인해 각 청크마다 I/O 블로킹
- 디스크 I/O가 병목이 될 수 있음

**해결 방안:**
```
현재: 청크 1 쓰기 → 기다림 → 청크 2 쓰기 → ...
최적화: 청크 버퍼 (10개) → 배치 쓰기 → 플러시
```

**구현:**
- 수신한 청크를 메모리 버퍼에 저장 (5-10개)
- 버퍼가 가득 차면 배치로 OPFS에 쓰기
- 네트워크 대기 중에 디스크 I/O 처리

**기대 효과:**
- 처리량: **10-20% 향상**
- CPU 사용률 감소

**구현 난이도:** ⭐⭐⭐ (중간)

---

### [4] 메모리 버퍼 풀 (버퍼 재사용) (🟡 중간 우선순위)

**현재 문제점:**
- 매 청크마다 새로운 버퍼 할당/해제
- GC(Garbage Collection) 압박으로 인한 지연

**해결 방안:**
```javascript
// 버퍼 풀 구현
class BufferPool {
  constructor(size = 100, chunkSize = 128 * 1024) {
    this.pool = Array(size).fill(null).map(() => 
      new Uint8Array(chunkSize)
    );
  }
  
  acquire() { return this.pool.pop(); }
  release(buffer) { this.pool.push(buffer); }
}
```

**기대 효과:**
- 메모리 할당 시간: **50% 감소**
- GC 일시 중지 횟수 감소

**구현 난이도:** ⭐⭐ (낮음)

---

### [5] 파일 압축 전송 (선택사항) (🟢 낮은 우선순위)

**현재 문제점:**
- 압축 가능한 파일(텍스트, 이미지 등)도 그대로 전송
- 네트워크 대역폭 낭비

**해결 방안:**
- Sender Worker에서 온-더-플라이 압축
- Receiver Worker에서 온-더-플라이 압축 해제
- 사용 라이브러리: fflate (이미 package.json에 있음)

**기대 효과:**
- 일반 파일: **20-40% 전송량 감소**
- 이미 압축된 파일(zip, jpg): 효과 없음

**구현 난이도:** ⭐⭐⭐⭐ (높음)

**트레이드오프:**
- CPU 사용률 증가 (압축/해제)
- 구현 복잡도 높음

---

### [6] 사전 버퍼링 (Read-Ahead) (🟡 중간 우선순위)

**현재 문제점:**
- 요청 후 읽기 시작하므로 지연 발생
- 네트워크 대기 중 디스크 읽기 안 함

**해결 방안:**
```
백그라운드에서 미리 읽기:
네트워크 전송 ─→ 동시에 ─→ 다음 청크 미리 읽기
```

**기대 효과:**
- 전송 지연: **5-10% 감소**
- 네트워크 활용률 향상

**구현 난이도:** ⭐⭐ (낮음)

## 🎯 추천 구현 순서

### Phase 1: 즉시 적용 (1-2주)
1. **동적 청크 크기 최적화** (RTT 기반)
2. **메모리 버퍼 풀** (GC 압박 감소)
3. **사전 버퍼링** (Read-Ahead)

**기대 성능 향상:** 20-30%

### Phase 2: 중기 (2-4주)
1. **OPFS 배치 쓰기** 최적화
2. **멀티 채널 병렬 전송** (2개 채널부터 시작)

**기대 성능 향상:** 50-100% (추가)

### Phase 3: 장기 (1-2개월)
1. **파일 압축 전송** (선택사항)
2. **적응형 멀티 채널** (4-8 채널)

**기대 성능 향상:** 100-200% (추가)

## 📈 성능 측정 지표

### 모니터링할 메트릭
- **처리량 (MB/s)**: 현재 전송 속도
- **버퍼 상태**: WebRTC 버퍼 크기 (Buffered Amount)
- **RTT**: 왕복 지연시간 (ms)
- **패킷 손실률**: 실패한 청크 비율
- **메모리 사용량**: Worker 메모리 (MB)
- **CPU 사용률**: 웹 워커 CPU 점유

### 측정 구현
```typescript
// webRTCService.ts에 추가
interface PerformanceMetrics {
  throughput: number;      // MB/s
  bufferedAmount: number;  // bytes
  avgRTT: number;         // ms
  packetLossRate: number; // 0-1
  memoryUsage: number;    // MB
  cpuUsage: number;       // 0-100
}
```

## ⚠️ 주의사항

### 메모리 제약
- 멀티 채널: 채널당 ~10-20MB 버퍼 필요
- 4개 채널 = 40-80MB 추가 메모리
- 모바일 기기에서는 2개 채널로 시작 권장

### 브라우저 호환성
- WebRTC 멀티 채널: 모든 최신 브라우저 지원
- fflate 압축: IE 미지원

### 네트워크 조건
- 불안정한 네트워크: 멀티 채널로 신뢰성 향상
- 고지연 네트워크: 동적 청크 크기로 적응
